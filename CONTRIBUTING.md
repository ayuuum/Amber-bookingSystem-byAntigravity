# Amber 開発への参加（Contributing）

Amber プロジェクトへようこそ！このドキュメントでは、共同作業者がスムーズかつ専門的に開発を進めるための基準とワークフローについて説明します。

## 🤝 開発フィロソフィー

- **PRD 第一主義**: 定義したものを作ります。仕様変更が必要な場合は、まず PRD を更新してください。
- **AI ネイティブ**: AI による開発支援（Antigravity など）を積極的に活用しつつ、人間が読んでも高品質なコード基準を維持します。
- **ユーザー中心**: すべての機能は、日本の現場作業者の課題を解決するためのものであるべきです。

## 🛠️ 開発ワークフロー

### 1. ブランチ命名規則
- `feature/機能名` (例: `feature/line-auth`)
- `fix/バグ概要` (例: `fix/booking-overlap`)
- `docs/ドキュメント更新` (例: `docs/update-api-spec`)

### 2. 開発の流れ
- 常に `main` ブランチから新しいブランチを作成してください。
- コミットは小さく、具体的で、わかりやすい説明を心がけてください。
- すべての変更は Pull Request (PR) を通じて提出してください。

### 3. データベースの変更 (Supabase)
スキーマの変更には Supabase Migrations を使用します。
- Supabase ダッシュボードで直接スキーマを変更 **しないでください**。
- Supabase CLI を使用してマイグレーションを作成してください: `supabase migration new migration_name`
- 新しいテーブルには必ず RLS (Row Level Security) を適用してください。

## 🎨 コーディング基準

### TypeScript
- Strict モードが有効です。`any` の多用を避けてください。
- フックを使用した関数コンポーネントを使用してください。
- プロップスやデータ構造には必ずインターフェース・型定義を行ってください。

### UI/デザイン
- 原則として **Tailwind CSS** を使用します。
- [design_guidelines.md](file:///Users/ayumu/Amber-House-PJ/docs/design_guidelines.md) のデザインシステムに従ってください。
- `shadcn/ui` を基本コンポーネントとして活用してください。

## 🧪 プルリクエスト (PR) プロセス

1. **セルフレビュー**: 変更内容が PRD と一致しているか確認してください。
2. **ビルドチェック**: ローカルで `npm run lint` と `npm run build` が通ることを確認してください。
3. **説明文**: 「何を」「なぜ」変更したかを具体的に記載し、関連する PRD のセクションをリンクしてください。
4. **承認**: `main` ブランチへのマージには、少なくとも 1 名の承認が必要です。
